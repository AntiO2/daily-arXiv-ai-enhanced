### [1] [PARNESS: A Paper Harness for End-to-End Automated Scientific Research with Dynamic Workflows, Full-Text Indexing, and Cross-Run Knowledge Accumulation](https://arxiv.org/abs/2605.05258)
*Yuchen Wang,Zhongzhi Luan*

Main category: cs.SE

TL;DR: 现有自主研究系统（AI-Scientist等）框架僵化，PARNESS通过DAG调度、全文PDF解析、知识图谱索引与编码代理扩展四个设计，实现了跨学科、跨运行的开源研究自动化框架。


<details>
  <summary>Details</summary>
Motivation: 现有LLM自主研究系统（AI-Scientist、PaperOrchestra等）虽能完成从构思到写论文的流程，但都在框架层面固定了单一的控制流形态（线性流水线、状态机、单代理循环或固定技能包），无法适应不同学科（实验、调研、仿真、理论）的动态工作流差异。

Method: PARNESS 采用四个设计要点：(i) 轻量DAG内核 + 四字段Agent契约，将调度与领域语义解耦，学科工作流可用YAML表达；(ii) 全文PDF解析与文献库子系统，将论文正文、图表索引为类型化对象，支持摘要回退；(iii) 知识图谱索引覆盖论文、想法、实验与代码仓库，支持场景类型化检索（相似/矛盾/跨领域/反直觉）；(iv) 通过小扩展接口允许现代编码代理（Claude Code等）增删模块。

Result: PARNESS 是首个结合声明式流水线、全文PDF与代码仓库索引、跨运行知识持久化的开源研究自动化系统，代码已公开在 GitHub。

Conclusion: 通过DAG内核解耦调度与语义、全文索引补齐论文正文信息、知识图谱实现跨运行知识复用、以及编码代理扩展机制，PARNESS 解决了现有框架僵化的五大根源问题，为多学科自主研究提供了灵活的开源基础设施。

Abstract: Recent autonomous research systems -- AI-Scientist, PaperOrchestra, AutoSOTA, DeepResearch, InternAgent, ResearchAgent and others -- show LLM agents can ideate, run experiments and write papers, but each fixes a particular control-flow shape (linear pipeline, state machine, single-agent loop, or fixed-recipe skill pack) at the framework level. We argue this rigidity has five roots: (1) workflows are dynamic and discipline-specific (lab work, surveys, simulations, theory all loop differently); (2) ideation is bounded by LLM context and cross-domain ideation needs knowledge a single context cannot hold; (3) summary-only views miss the paper body, yet full-text access is uneven, so the cumulative corpus must do the work; (4) a paper's open-source repository is often the only complete specification of its experimental scheme, but the paper-to-code link is neglected; (5) no tool persists cross-run knowledge retrievably into a finite LLM context.
  We present PARNESS, an open-source framework built on four design moves. (i) A thin DAG kernel with a four-field Agent contract decouples scheduling from domain semantics, so any discipline's loop is expressible as user-editable YAML. (ii) A full-text PDF-parsing and literature-library subsystem indexes paper bodies, figures and tables as typed objects, with graceful abstract-only fall-back. (iii) A knowledge-graph index over papers, ideas, experiments and code repositories, with scenario-typed retrieval (similar / contradictory / cross-domain / counter-intuitive), surfaces a focused slice into each LLM call. (iv) A small extension surface lets any modern coding agent (Claude Code, Cursor, Copilot, OpenCode) add or replace any module. To our knowledge PARNESS is the first open-source system combining declarative pipelines, full-PDF and code-repository indexing, and cross-run knowledge. Source: https://github.com/gtrhythm/PARNESS

### [2] [An Extensible and Verifiable Language for Query Rewrite Rules](https://arxiv.org/abs/2605.05536)
*Sicheng Pan,Shuxian Wang,Wesley Zheng,Zirong Zeng,Vijay Sharma,Alvin Cheung*

Main category: cs.DB

TL;DR: 提出Rulescript，一种与执行引擎无关的领域特定语言，用于定义查询重写规则，实现"一次编写，到处部署"的查询计划重写范式。


<details>
  <summary>Details</summary>
Motivation: 现有查询重写规则通常手动实现、与特定执行引擎紧耦合且缺乏形式化正确性保证，导致开发新引擎时需重复实现规则，工程成本高、可移植性差且易引入错误。

Method: 设计Rulescript DSL，基于关系代数核心语言将规则分解为匹配和变换两个阶段；开发者通过核心操作符进行模式匹配并构建语义等价的变换计划；支持自定义操作符扩展；通过轻量级适配器与目标引擎集成；所有重写自动进行形式化正确性验证。

Result: 用Rulescript重写了Apache Calcite的33条重写规则，并扩展了多个自定义操作符；自动部署到CockroachDB和Apache Data Fusion两个不同后端引擎，验证了可移植性。

Conclusion: Rulescript实现了查询计划重写的"一次编写，到处部署"范式，以最小代价将已有规则部署到新数据引擎上。

Abstract: Logical query plan rewriting transforms a relational database query into an equivalent but more efficient form and is crucial to the performance of database-backed applications. In existing systems, rewrite rules are typically implemented manually, tightly coupled to specific execution engines, and often lack formal correctness guarantees. Consequently, developing a new engine requires reimplementing both legacy and new rules, incurring significant engineering cost, limiting portability, and every new implementation is an opportunity for introducing new bugs.
  We introduce Rulescript, an engine-agnostic domain-specific language (DSL) for developing query rewrite rules. Rulescript separates rule definition from execution infrastructure via a relational algebra-inspired core language and an explicit decomposition of rules into matching and transformation phases. Developers express rewrites by pattern-matching query plans using Rulescript's core operators and constructing semantically equivalent transformed plans, with all rewrites automatically verified formally to ensure correctness. Rulescript is extensible: users can define custom operators in terms of the core language to capture engine-specific semantics. To integrate with an existing system, developers need only implement a lightweight adapter that maps Rulescript's core and custom operators to the operators implemented in the target engine.
  We evaluate Rulescript by reimplementing 33 rewrite rules from Apache Calcite and extending the language with several custom operators. To demonstrate portability, we automatically deploy these rules to CockroachDB and Apache Data Fusion, two engines with substantially different backends. Our results show that Rulescript enables "write once, deploy everywhere" paradigm for query plan rewriting, with minimal effort required to deploy previously written rules on a new data engine.
